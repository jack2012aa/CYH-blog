[SyntaxNext] Project Introduction

Jul 7, 2025

Hi everyone. This is a new series about what I learned while developing our side project, SyntaxNext.

What SyntaxNext Does

SyntaxNext is an AI-powered mock interview agent. It works as a Chrome extension, and users can choose any LeetCode problem. During the interview, the extension shows the stages and tasks that an interviewee should complete. The agent will describe the question, and users should tell the agent how they plan to solve the problem. Users can go to the next step only when the agent thinks they have finished the required tasks in the current step.

After users finish their code and the agent approves the solution, they can submit their code to LeetCode, just like we always do. If everything goes well and the interview is done, users receive an evaluation of their performance.

Project Architecture

We did some research before starting development and found no similar products on the web. Most mock interview services are for behavioral interviews or other types.

Developing this project was harder than we predicted. For example, writing a functional content script requires a full understanding of LeetCode’s behavior in different components. We divided the project into four components/microservices:

  1. extension,
  2. agent,
  3. speech-to-text, and
  4. login. Except for the extension, these components can be reused for new types of user interfaces.

Chrome Extension

Based on Manifest V3, a Chrome extension is composed of:

  1. content script(s): script(s) that are inserted into specific pages,
  2. offscreen(s): invisible pages,
  3. service worker: a temporary worker for the extension,
  4. popup: a popup page,
  5. manifest: a settings file.

An extension can only contain some of these components. For example, we only use 1, 3, and 5.

User Media

Our agent interacts with users by voice. Users’ voices are processed by a speech-to-text (STT) service. To capture the voice, we have to get permission to use the device’s microphone, which can be done by the getUserMedia interface. However, if it is called on a LeetCode page, what users will see is “LeetCode is asking for the permission,” which is not what we want.

To create a warning popup with our application’s name, we have to create a page whose source is the Chrome extension itself. The popup meets the requirement, but we want it to be invisible. On the other hand, although an offscreen document is invisible, a warning in it won’t appear on the current user page. Eventually, we used an iframe inserted by the content script. The iframe is invisible, and because of Chrome’s logic, its source is the extension.

WebSocket Connection

In the early design, the agent could send a message to the extension when the user was not speaking for a long time to check what the user was thinking. This feature was postponed and, in fact, can also be satisfied by a traditional HTTP request (by setting a timeout in the frontend). Furthermore, the requirements of STT pushed us to use WebSockets.

But where should we build the connection? In the official documentation, Google suggests using the service worker. However, the service worker could be unloaded if no event happens for a period of time. The inserted iframe seems like the best choice.

(Actually, the WebSocket to the agent is unnecessary. LangGraph can be stateless by using a thread ID and state. Hence, RESTful APIs are totally practicable and would obtain higher scalability. Nevertheless, changing the current implementation is too time-consuming, and there aren’t that many users for now.)

Interaction with LeetCode

Since the extension is based on LeetCode, it has to get information from the page and change some behavior. The three most noteworthy cases are code pad fetching, submit button rewriting, and tab replacement.

The straightforward method to get the code pad content is to read the HTML content. But because the code pad component that LeetCode uses (Monaco) only creates elements that are shown on screen, fetching long code is hard. Though Monaco provides an API to get the code content, the runtime of the content script and LeetCode’s JavaScript is different, so we can’t get the Monaco object. The solution is to insert a script element into the LeetCode HTML. The script listens to window messages, fetches the code, and then replies once there is a request.

The redirection after code submission is another issue. It breaks the integrity of the user experience. The Chrome extension provides several ways to block redirection, but they don’t work on those invoked by React. We have to write a new listener that submits, polls the result, and notifies the agent when clicked.

LeetCode uses React on its pages, which prevents us from appending a new tab to the current layout. So, we have to accept the second plan: changing the content of an existing tab. We chose the Solution tab. Surprisingly, React doesn’t change the content back even if we focus on another tab and then go back. This is an interesting insight, but since I haven’t done research on it, we can make this a future topic.
